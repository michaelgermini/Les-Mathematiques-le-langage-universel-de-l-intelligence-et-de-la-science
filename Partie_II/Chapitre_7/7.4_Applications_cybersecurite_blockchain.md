# 7.4. Applications : Cybersécurité, Algorithmes, Blockchain

## Introduction

Les mathématiques discrètes — graphes, logique, combinatoire, cryptographie — trouvent leurs applications les plus spectaculaires dans le monde numérique. Ce chapitre explore trois domaines où ces outils mathématiques sont indispensables : la cybersécurité, l'algorithmique et la blockchain.

---

## 1. Cybersécurité : protéger le monde numérique

### 1.1 Authentification

#### Authentification par mot de passe
- **Stockage sécurisé** : jamais en clair, toujours haché
- **Sel (salt)** : valeur aléatoire unique par utilisateur
- **Fonctions adaptées** : bcrypt, scrypt, Argon2 (coûteuses en calcul)

```
hash_stocké = Argon2(mot_de_passe || sel, paramètres_coût)
```

#### Authentification multi-facteurs (MFA)
Combine plusieurs facteurs :
| Facteur | Type | Exemple |
|---------|------|---------|
| **Connaissance** | Ce que vous savez | Mot de passe, PIN |
| **Possession** | Ce que vous avez | Téléphone, token TOTP |
| **Inhérence** | Ce que vous êtes | Empreinte, visage, iris |

#### TOTP (Time-based One-Time Password)
```
TOTP = HMAC-SHA1(clé_secrète, temps/30) mod 10^6
```
Génère un code à 6 chiffres valide 30 secondes (Google Authenticator, etc.)

### 1.2 Chiffrement des données

#### Au repos (data at rest)
- **Chiffrement de disque** : BitLocker (Windows), LUKS (Linux), FileVault (macOS)
- **Chiffrement de fichiers** : VeraCrypt, age, GPG
- **Bases de données** : TDE (Transparent Data Encryption)

#### En transit (data in motion)
- **TLS/HTTPS** : chiffrement des communications web
- **VPN** : tunnel chiffré (WireGuard, OpenVPN, IPsec)
- **SSH** : accès distant sécurisé

#### De bout en bout (end-to-end)
- **Signal Protocol** : Double Ratchet (Signal, WhatsApp)
- **PGP/GPG** : emails chiffrés
- Même le fournisseur ne peut pas lire les messages

### 1.3 Infrastructure à clé publique (PKI)

#### Composants
- **Autorité de certification (CA)** : émet les certificats
- **Autorité d'enregistrement (RA)** : vérifie les identités
- **Certificat X.509** : lie identité et clé publique
- **Liste de révocation (CRL)** / **OCSP** : certificats invalidés

#### Chaîne de confiance
```
CA Racine (auto-signé, dans le navigateur)
    └── CA Intermédiaire (signé par racine)
            └── Certificat serveur (signé par intermédiaire)
```

#### Certificate Transparency
- Logs publics de tous les certificats émis
- Détection des certificats frauduleux
- Obligatoire pour Chrome depuis 2018

### 1.4 Sécurité des applications

#### OWASP Top 10 (vulnérabilités web)
1. Broken Access Control
2. Cryptographic Failures
3. Injection (SQL, XSS, etc.)
4. Insecure Design
5. Security Misconfiguration
6. Vulnerable Components
7. Authentication Failures
8. Data Integrity Failures
9. Logging Failures
10. Server-Side Request Forgery

#### Défenses
- **Validation des entrées** : ne jamais faire confiance aux données utilisateur
- **Requêtes préparées** : contre l'injection SQL
- **CSP** : Content Security Policy contre XSS
- **CORS** : contrôle des requêtes cross-origin

## 2. Algorithmes et structures de données

### 2.1 Algorithmes de graphes en pratique

#### Navigation et routage
- **Dijkstra / A*** : GPS, navigation (Google Maps, Waze)
- **Contraction Hierarchies** : pré-calcul pour requêtes ultra-rapides
- **Routage dynamique** : prise en compte du trafic en temps réel

#### Réseaux sociaux
- **BFS** : degrés de séparation, suggestions d'amis
- **PageRank** : importance des nœuds
- **Communautés** : algorithme de Louvain, Label Propagation

#### Optimisation
- **Flot maximal** : affectation, transport
- **Couplage** : matching biparti (emplois, organes)
- **TSP** : tournées de livraison (heuristiques, métaheuristiques)

### 2.2 Structures de données avancées

#### Tables de hachage
- **Complexité moyenne** : O(1) pour insertion, recherche, suppression
- **Gestion des collisions** : chaînage, adressage ouvert
- **Applications** : dictionnaires, caches, déduplication

#### Arbres équilibrés
| Structure | Opérations | Usage |
|-----------|------------|-------|
| **AVL** | O(log n) | Recherche fréquente |
| **Rouge-noir** | O(log n) | Usage général (std::map) |
| **B-tree** | O(log n) | Bases de données, systèmes de fichiers |
| **B+ tree** | O(log n) | Index de bases de données |

#### Structures probabilistes
- **Bloom filter** : test d'appartenance (faux positifs possibles)
- **Count-Min Sketch** : estimation de fréquences
- **HyperLogLog** : estimation de cardinalité

### 2.3 Complexité algorithmique

#### Classes de complexité

```
P ⊆ NP ⊆ PSPACE ⊆ EXPTIME
```

| Classe | Définition | Exemple |
|--------|------------|---------|
| **P** | Résolu en temps polynomial | Tri, plus court chemin |
| **NP** | Vérifié en temps polynomial | SAT, TSP, Sudoku |
| **NP-complet** | Les plus durs de NP | SAT, 3-SAT, Clique |
| **NP-difficile** | Au moins aussi dur que NP-complet | Problème d'arrêt |

#### Théorème de Cook-Levin
SAT (satisfiabilité booléenne) est NP-complet.
Tout problème NP peut être réduit à SAT en temps polynomial.

#### Implications pratiques
- Problèmes NP-complets : pas d'algorithme polynomial connu
- Solutions : heuristiques, approximations, algorithmes paramétrés
- Cryptographie : sécurité basée sur des problèmes supposés difficiles

### 2.4 Algorithmes d'approximation

Pour les problèmes NP-difficiles :

| Problème | Approximation | Garantie |
|----------|---------------|----------|
| **Vertex Cover** | Glouton | 2-approximation |
| **TSP métrique** | Christofides | 1.5-approximation |
| **Set Cover** | Glouton | O(log n)-approximation |
| **Knapsack** | FPTAS | (1+ε)-approximation |

## 3. Blockchain et systèmes distribués

### 3.1 Fondements de la blockchain

#### Structure
```
Bloc N-1          Bloc N            Bloc N+1
┌─────────┐      ┌─────────┐      ┌─────────┐
│ Hash(N-2)│ ←── │ Hash(N-1)│ ←── │ Hash(N) │
│ Données │      │ Données │      │ Données │
│ Nonce   │      │ Nonce   │      │ Nonce   │
│ Hash(N-1)│      │ Hash(N) │      │ Hash(N+1)│
└─────────┘      └─────────┘      └─────────┘
```

- **Chaînage cryptographique** : chaque bloc contient le hash du précédent
- **Immuabilité** : modifier un bloc invalide tous les suivants
- **Registre distribué** : copies sur tous les nœuds du réseau

#### Composants cryptographiques
- **Fonctions de hachage** : SHA-256, Keccak (intégrité)
- **Signatures numériques** : ECDSA, Ed25519 (authentification)
- **Arbres de Merkle** : vérification efficace des transactions

### 3.2 Mécanismes de consensus

#### Preuve de travail (Proof of Work - PoW)
```
Trouver nonce tel que Hash(bloc || nonce) < cible
```
- **Difficulté** : ajustée pour maintenir un temps de bloc constant
- **Sécurité** : attaque 51% requiert la majorité de la puissance de calcul
- **Inconvénient** : consommation énergétique massive

#### Preuve d'enjeu (Proof of Stake - PoS)
- **Principe** : validateurs choisis proportionnellement à leur mise
- **Slashing** : pénalité pour comportement malveillant
- **Avantage** : ~99% moins énergivore que PoW

#### Autres mécanismes
| Consensus | Principe | Exemple |
|-----------|----------|---------|
| **DPoS** | Délégation à des représentants élus | EOS, Tron |
| **PBFT** | Tolérance aux fautes byzantines | Hyperledger |
| **PoA** | Autorités identifiées | Réseaux privés |
| **PoH** | Preuve d'historique (horodatage) | Solana |

### 3.3 Cryptomonnaies

#### Bitcoin
- **Première blockchain** : 2009, Satoshi Nakamoto
- **Modèle UTXO** : Unspent Transaction Outputs
- **Script** : langage de script limité (non Turing-complet)
- **Offre limitée** : 21 millions de BTC maximum

#### Ethereum
- **Smart contracts** : programmes exécutés sur la blockchain
- **EVM** : Ethereum Virtual Machine (Turing-complet)
- **Gas** : unité de mesure du coût computationnel
- **Transition PoS** : The Merge (2022)

#### Stablecoins
- **Collatéralisés** : USDC, USDT (réserves en dollars)
- **Algorithmiques** : mécanismes de stabilisation automatiques
- **Crypto-collatéralisés** : DAI (sur-collatéralisation en ETH)

### 3.4 Smart contracts et DeFi

#### Smart contracts
```solidity
// Exemple simplifié (Solidity)
contract SimpleStorage {
    uint256 private value;
    
    function set(uint256 _value) public {
        value = _value;
    }
    
    function get() public view returns (uint256) {
        return value;
    }
}
```

- **Déterministes** : même entrée → même sortie
- **Immuables** : code déployé non modifiable (sauf patterns upgradeable)
- **Composables** : contrats peuvent appeler d'autres contrats

#### Finance décentralisée (DeFi)
| Protocole | Fonction | Exemple |
|-----------|----------|---------|
| **DEX** | Échange décentralisé | Uniswap, Curve |
| **Lending** | Prêt/emprunt | Aave, Compound |
| **Staking** | Rendement sur dépôts | Lido |
| **Bridges** | Transfert cross-chain | Wormhole |

### 3.5 Applications au-delà des cryptomonnaies

#### Supply chain
- **Traçabilité** : origine et parcours des produits
- **Authenticité** : lutte contre la contrefaçon
- **Exemples** : IBM Food Trust, VeChain

#### Identité numérique
- **Self-Sovereign Identity (SSI)** : contrôle par l'utilisateur
- **Verifiable Credentials** : attestations vérifiables
- **DID** : Decentralized Identifiers

#### NFTs et actifs numériques
- **Non-Fungible Tokens** : certificats de propriété uniques
- **Art numérique** : provenance et authenticité
- **Gaming** : objets in-game transférables

#### Vote électronique
- **Transparence** : résultats vérifiables publiquement
- **Défis** : secret du vote, coercition, accessibilité

## 4. Convergences et perspectives

### 4.1 Zero-Knowledge Proofs (ZKP)

Prouver une affirmation sans révéler d'information supplémentaire.

#### Applications
- **Privacy coins** : Zcash (zk-SNARKs)
- **Scaling** : zk-Rollups (zkSync, StarkNet)
- **Authentification** : prouver son âge sans révéler sa date de naissance

#### Types
- **zk-SNARKs** : Succinct Non-interactive Arguments of Knowledge
- **zk-STARKs** : Scalable Transparent Arguments of Knowledge
- **Bulletproofs** : preuves de range sans trusted setup

### 4.2 Calcul multipartite sécurisé (MPC)

Calculer une fonction sur des données privées sans les révéler.

- **Portefeuilles MPC** : clé privée distribuée
- **Enchères scellées** : offres secrètes
- **Statistiques privées** : analyses sur données sensibles

### 4.3 Chiffrement homomorphe

Calculs sur données chiffrées sans les déchiffrer.

- **Applications** : cloud computing privé, ML sur données sensibles
- **Limitations** : encore très lent (amélioration rapide)

## Conclusion

Les mathématiques discrètes sont le fondement invisible de notre infrastructure numérique :

1. **La cybersécurité** protège nos données grâce à la cryptographie et aux protocoles rigoureux
2. **L'algorithmique** optimise les systèmes qui font tourner le monde numérique
3. **La blockchain** crée de nouveaux paradigmes de confiance décentralisée

Ces domaines évoluent rapidement, portés par des avancées mathématiques (preuves à divulgation nulle, cryptographie post-quantique) et des défis sociétaux (vie privée, souveraineté numérique, décentralisation).
